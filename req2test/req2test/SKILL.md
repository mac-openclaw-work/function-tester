# req2test - 需求文档转功能测试点

## 快速开始

### 输出格式
- **Word 文档**：优先使用 `python3 -c "import docx; ..."` 读取 `.docx` 文件
- **Markdown**：直接读取文本内容
- **纯文本**：直接解析段落结构

### 输出格式
输出为 Markdown 无序列表，每个测试点包含优先级和模块标签，格式：
- `[P0/P1/P2/P3] [模块名] 测试点描述`

#### 模块标注规则

**模块识别**：测试点标题中` - `分隔的第一部分即为模块名，模块名应精简到2-6个字。

**模块命名规范**：
- 使用文档中明确的功能模块名称
- 多级模块用`/`分隔，如 `OpenClaw实例/部署配置`
- 无明确模块时，从操作主体推断（如"用户操作"→"用户"）

#### 优先级定义

| 优先级 | 含义 | 判断依据 |
|--------|------|---------|
| **P0** | 核心功能阻塞 | 主流程必经步骤，失败会导致后续无法进行 |
| **P1** | 重要功能 | 核心功能点，但有替代方案或非必经路径 |
| **P2** | 一般功能 | 重要但非核心的功能验证点 |
| **P3** | 次要功能 | 边界情况、异常处理、易用性验证 |

#### 优先级判断规则

1. **操作步骤中的关键节点** → P0/P1
   - 点击"部署"、"确定"、"提交"等最终确认操作 → P0
   - 配置核心参数（选择实例、填写必填项） → P1

2. **预期结果中的验证点** → P0/P1
   - "状态变为正常"、"创建成功" → P0
   - "正确显示"、"正常响应" → P1

3. **非必经路径/异常处理** → P2/P3
   - 修改操作（非创建）、查看详情 → P2
   - 边界条件、异常提示 → P3

## 转换规则

### 1. 识别需求文档结构

需求文档通常包含以下部分，按优先级提取：

| 部分 | 说明 | 提取策略 |
|------|------|---------|
| **测试目的/场景描述** | 功能要解决什么问题 | 提取核心验证点 |
| **操作流程/步骤** | 用户如何操作 | 拆解为独立操作节点 |
| **功能描述** | 功能的具体行为 | 提取可验证的行为点 |
| **角色类型** | 不同用户的操作权限 | 按角色拆分测试点 |
| **预置条件** | 测试前提 | 转化为前置验证测试点 |
| **预期结果** | 期望的系统行为 | 对应验证断言点 |

### 2. 测试点生成原则

**拆分到最小可测试单元**：
- 每个测试点应独立可执行
- 一个测试点验证一个明确的行为/结果
- 避免多个验证点混杂在一个测试项中

**测试点命名规范**：
- 格式：`[功能模块] [操作/验证] [预期结果]`
- 示例：`OpenClaw实例部署 - 选择数据中心和资源池 - 成功关联`
- 示例：`数字员工创建 - 配置大模型接入 - API连接成功`

### 3. 按场景分类组织

生成的测试点按文档中的**测试子项**或**功能模块**分组，用 `##` 二级标题区分。

## 处理流程

1. **解析文档**：读取需求文档内容，识别章节结构
2. **定位测试场景**：找到测试目的、测试步骤、预期结果等关键部分
3. **逐项提取**：按照转换规则生成测试点
4. **模块标注**：为每个测试点标注所属模块
5. **去重合并**：合并重复或类似的测试点
6. **统计汇总**：生成模块分布统计表
7. **格式输出**：先输出模块索引，再按场景分组输出测试点

---

## 注意事项

- 如果文档中包含表格形式的测试步骤，需要解析表格行并提取为独立测试点
- 对于复杂的操作流程（如步骤1.2.3），将每个子步骤拆分为独立测试点
- 角色类型不同会导致不同操作权限，需为每个角色生成对应的测试点
- 预置条件中的每一项都可转化为一个"环境验证测试点"

如需查看完整解析逻辑和文档结构参考，见 [references/parsing.md](references/parsing.md)。
