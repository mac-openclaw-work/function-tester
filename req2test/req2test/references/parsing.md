# 需求文档解析参考

## 支持的文档格式

### Word (.docx)
```python
from docx import Document
doc = Document("需求文档.docx")
for para in doc.paragraphs:
    print(para.text)
```

### Markdown
直接读取文件内容，按 `##` 一级标题和 `###` 二级标题识别结构。

---

## 关键章节识别

需求文档中用于提取测试点的关键章节：

| 章节关键词 | 说明 | 提取内容 |
|-----------|------|---------|
| 测试目的 / 测试场景 | 描述功能要验证什么 | 核心验证点 |
| 测试步骤 / 操作步骤 | 具体操作流程 | 按步骤拆解 |
| 预置条件 / 前置条件 | 前提环境要求 | 前置验证测试点 |
| 预期结果 / 期望结果 | 期望的系统行为 | 对应验证断言点 |
| 角色类型 / 用户类型 | 不同用户的操作差异 | 按角色拆分 |
| 功能描述 | 功能的具体行为 | 可验证的行为点 |

## 测试点生成模式

### 模式1：操作步骤 → 测试点
对于"步骤1. → 步骤2. → 步骤3."这种流程：
- 每个步骤转化为一个测试点
- 命名格式：`[模块] [操作] - [预期]`

### 模式2：角色 + 操作 → 测试点
对于涉及多角色的需求，为每个角色生成对应测试点：
- 命名格式：`[角色] [操作] [资源] - [预期]`

### 模式3：功能点 → 测试点
对于功能列表，按功能点逐个生成测试点。

## 优先级判断规则

| 优先级 | 含义 | 判断依据 |
|--------|------|---------|
| **P0** | 核心功能阻塞 | 主流程必经步骤，失败会导致后续无法进行 |
| **P1** | 重要功能 | 核心功能点，但有替代方案或非必经路径 |
| **P2** | 一般功能 | 重要但非核心的功能验证点 |
| **P3** | 次要功能 | 边界情况、异常处理、易用性验证 |

### 判断逻辑

1. **操作步骤中的关键节点**
   - 点击"部署"、"确定"、"提交"等最终确认操作 → P0
   - 配置核心参数（选择实例、填写必填项） → P1

2. **预期结果中的验证点**
   - "状态变为正常"、"创建成功" → P0
   - "正确显示"、"正常响应" → P1

3. **非必经路径/异常处理**
   - 修改操作（非创建）、查看详情 → P2
   - 边界条件、异常提示 → P3

## 模块标注规则

测试点使用 `[模块名]` 标注所属功能模块。

### 模块识别策略

1. **文档结构映射**：文档中的"测试子项"即为一级模块
   - "OpenClaw实例部署与数字员工创建端到端测试" → 模块：`OpenClaw实例部署`、`数字员工创建`、`网络隔离验证`

2. **操作主体推断**：无明确模块时，按操作主体命名
   - "用户操作" → `用户`
   - "系统配置" → `系统配置`

3. **命名规范**：
   - 精简到2-6个字
   - 多级模块用`/`分隔，如 `OpenClaw/实例部署`
   - 保持文档中的原始模块名称优先

### 模块分布统计

生成测试点后，统计各模块的测试点数量和优先级分布：

| 模块 | 测试点数量 | P0 | P1 | P2 | P3 |
|------|-----------|----|----|----|----|
| OpenClaw实例部署 | 9 | 2 | 5 | 2 | 0 |
| ... | ... | ... | ... | ... | ... |

## 去重合并策略

1. **步骤级重复**：同一操作的不同表述 → 保留最完整的
2. **预期结果重复**：同一结果的不同描述 → 合并为一条
3. **角色重复**：同一角色的不同操作 → 合并到同一角色下
